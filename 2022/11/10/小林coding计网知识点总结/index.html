小林coding计网总结

主要记录一些自己之前不知道的或者重要的、易忘的知识点。根据小林的博客自己想到的面试高频题可能也会记录。

1. 基础篇

  1. TCP不会为没有数据的ACK超时重传，所以如果ACK报文丢失，ACK的接收方会重传自己的报文给发送方，直到收到ACK发送方返回的ACK。
  2. 不能两次握手：无法阻止历史连接的建立，造成资源浪费；如果服务端的SYN+ACK报文丢失，会造成客户端与服务端的初始序列号无法同步。
  3. 不四次握手：服务端的ACK与同步信号SYN明显可以合并
  4. 不能三次挥手：要三次挥手只能将被动方的ACK与FIN合并，但是在被动方收到ACK时可能还有数据要传，这种情况下不能立刻断开连接。
  5. MTU：一个网络包的最大长度，以太网中一般为1500字节（IP头部）
    MSS：网络包所能容纳TCP数据的最大长度（TCP的数据部分，不包括TCP头部）
  6. 网卡会在网络包的头部添加报头和起始帧分界符，还会在帧的尾部添加FCS帧校验序列。
  7. linux中用内核态中的sk_buff维护待发送的包。因为TCP协议支持丢失重传，所以sk_buff只是拷贝发送，直到收到ACK才删除。
    各个层数据包叫法不同但用同一个结构体维护的原因：多次拷贝降低CPU效率，所以通过调整sk_buff->data指针来添加与剥离头部。

2. HTTP

  1. content-length字段表明服务器回应的数据长度，是为了设置HTTP body的边界解决TCP粘包的问题。
  2. 安全：请求方法不会破坏服务器上的资源。
    幂等：多次执行相同的操作获得的结果相同。
    GET操作是安全且幂等的，因此可以对其请求的数据做缓存。
    POST操作既不安全也不幂等，因此浏览器一般不会缓存POST请求，也不能将其保存为书签。
    如果不按照RFC规范定义语义实现GET与POST，那GET也可能实现增删，POST也可能实现查。
  3. HTTP缓存分为强制缓存与协商缓存。强制缓存通过判断HTTP响应头部中的Cache-Control（相对时间，优先级高）和Expires（绝对时间）两个字段决定是否使用本地缓存；协商缓存流程如图：
  4. HTTP1.1：
    1. 优点：
      1. 简单：报文格式使用header+body，头部信息使用key-value，易于理解。
      2. 可拓展性强：HTTP协议的每个组成部分都允许开发人员自定义，且允许第七层之下的层随意变化。
      3. 应用广泛与跨平台。
    2. 双刃剑：
      1. 无状态：服务器不需要额外信息记录状态，减轻压力；如果完成有关联的操作时很麻烦，需要借助Cookie技术，在第一次相应时附加Cookie信息，之后根据接收报文的Cookie信息判断是否关联。
      2. 明文传输导致的不安全：通信内容不完整、通信对象伪装、通信内容被窃听。
    3. 特点：
      1. 长连接/持久连接
      2. 管道网络传输：允许连续发出报文而不必等待ACK；但是会造成队头阻塞，而且该功能默认关闭，实际上并没有使用。
    4. 优化方式：
      1. 尽量避免发送HTTP请求：缓存技术。
      2. 减少HTTP请求次数：
        1. 减少重定向请求次数：将重定向工作交由代理服务器完成。
        2. 合并请求：将小图片拼成大图片、打包资源等。
        3. 延迟发送请求：进入页面时只获取用户能看到的页面资源。
      3. 压缩：
        1. 无损压缩：用在文本文件、代码文件、可执行文件中，去除换行符、空格之后用霍夫曼编码压缩（gzip等）。
        2. 有损压缩：用在多媒体数据。
  5. HTTPS
    1. 特点：在TCP与HTTP层之间加入了SSL/TLS安全协议，在三次握手后还需要进行SSL/TLS握手。HTTPS实现了信息加密（防窃听）、校验机制（防篡改）与身份证书（放伪造）。
    2. SSL/TLS安全协议：使用了对称加密与非对称加密的混合加密。
      1. 对称加密：双方用同一把密钥加密与解密。但是密钥会在交换过程中被窃取。
      2. 非对称加密：接收方生成公钥与私钥，将公钥发给发送方。公钥加密的消息只有私钥能解密。但是速度慢。
      3. 混合加密——保证信息加密：
        1. 接收方生成公钥a和私钥，并发送给发送方公钥a。
        2. 发送方生成临时密钥公钥b作为对称加密的公钥，使用公钥b加密发送的信息，使用公钥a加密公钥b，并打包发送。
        3. 接收方使用私钥解密得到公钥b，然后用公钥b解密得到信息。
      4. 摘要算法+数字签名——防篡改：
        1. 摘要算法：发送方计算内容的哈希值，加密并发送给接收方，接收方接收后计算内容的哈希值并与解密的内容对比是否一致。
        2. 数字签名：使用非对称加密，私钥加密公钥解密，来确保发送者是私钥持有者。
      5. 数字证书——防伪造：
        1. CA（数字证书认证机构）。CA的公钥已经事先设置在操作系统中
        2. 流程：
          1. 服务器将自己的公钥上传到CA。
          2. CA打包持有者的公钥、用途、颁发者、有效时间等信息，计算Hash值，并用私钥给服务器的公钥签名。CA将签名添加在文件证书上，形成数字证书。
          3. 客户端接收到服务端的数字证书后，先用CA的公钥（操作系统内置）解密签名，并与自己算出的Hash值对照确认身份，并获得服务器的公钥。
          4. 客户端使用服务器的公钥加密发送。
          5. 服务器解密。
        3. 信任链：证书存在层级关系。如果收到的不是根证书，客户端会逐级向上请求证书，直到根证书，再从根证书逐级向下查询下级证书是否可信。
  6. TLS四次握手——RSA算法
    1. 第一次：客户端发送Client Hello消息，给出自己的TLS版本号、支持的密码套件列表与生成的随机数。
    2. 第二次：服务器确认是否支持TLS版本号，并选择一个密码套件，生成一个随机数，返回Server Hello消息。密码套件的基本形式是“密钥交换算法 + 签名算法 + 对称加密算法 + 摘要算法”。然后服务器发送含有数字证书的Server Certificate给客户端。最后，服务器发送Server Hello Done表示本次打招呼完毕。
    3. 第三次：客户端验证证书确认是否可信。客户端生成新的随机数，用服务器的RSA公钥加密并通过Client Key Exchange消息传递给服务器。服务器用RSA私钥解密得到随机数。双方使用三个随机数生成会话密钥，客户端发送Change Cipher Spec告诉服务端开始用加密方式发送消息，然后再发送Encrypted Handshake Message消息，将之前发送过的所有数据做个摘要并用会话密钥加密，让服务器验证加密通信是否可用，以及握手信息是否被篡改过。
    4. 第四次：服务器同样返回两种消息，双方验证无误后握手正式完成。
    5. 缺陷：不支持前向保密：一旦服务器的私钥泄露，过去被第三方结果的所有TLS通讯密文都会被破解。
  7. TLS四次握手——ECDHE算法
    1. 离散对数：即使知道底数a和真数b，在现有计算机的水平下几乎无法计算出离散对数。
    2. 发展历程：基于离散对数的DH算法，分为两种实现：static DH算法中一方私钥是静态的，这样会导致随着时间延长，黑客截获海量的密钥协商过程并暴力破解私钥；DHE算法中双方的私钥都随机临时生成，但是因为需要做大量乘法导致计算性能不佳。
    3. 原理：双方事先确定使用的椭圆曲线以及基点G；随机生成私钥d并与G相乘得到公钥Q；双方交换公钥，与自己的私钥相乘得到坐标x，y，x相同，即为共享密钥。
    4. 第一次：与RSA相同。
    5. 第二次：与RSA相似，但是选择ECDHE作为密钥协商算法。服务器发送Certificate消息后发送Server Key Exchange消息，包含服务器选择的椭圆曲线、基点与计算出来的服务器公钥，生成的私钥保留在本地。最后发送Server Hello Done。
    6. 第三次：客户端验证证书确认是否可信。客户端同样发送Client Key Exchange给服务端。双方计算出x，并使用客户端随机数+服务器随机数+x生成会话密钥。后续与RSA相同。
    7. 第四次：与RSA相同。
    8. 不同：使用ECDHE算法，客户端可以不等服务端最后一次TLS握手就能提前发送加密的HTTP数据，节省了一个消息往返的时间。
  8. HTTP1.1优化
    1. 硬件优化、软件优化：升级。
    2. TLS优化：升级到1.3。
      1. 把Hello和公钥交换合并成一个消息，将握手时间减少到1RTT。
      2. 减少密码套件，废除不支持前向安全的RSA与DH算法，防止降级攻击。
    3. 证书优化：
      1. 传输：选择椭圆曲线（ECDSA）证书而不是RSA证书，因为在相同安全强度下ECC密钥长度远小于RSA。
      2. 验证：
        1. 证书吊销列表CRL：CA定期更新维护被撤销信任的证书序号。实时性较差；随证书增多列表变大，下载速度变慢。
        2. 在线证书状态协议OCSP：向CA发送查询请求证书的有效状态。受制于网络状态与CA服务器繁忙。
        3. OCSP Stapling：服务器向CA周期性查询正梳妆台，获得带有时间戳和签名的响应结果并缓存。
    4. 会话复用：
      1. Session ID：首次握手后用Session ID标识会话密钥。服务器内存压力大；客户端不一定连接到同一台服务器。
      2. Session Ticket：服务器加密会话密钥作为Ticket发送给客户端缓存。不具备前向安全性；应对重放攻击困难。
      3. Pre-shared Key：TLS1.3，只需要0RTT就能恢复会话。将Ticket与HTTP请求一同发送给服务端。
      4. 应对重放攻击：设定合理的过期时间；只对安全的HTTP请求使用会话复用。
  9. HTTP2.0
    1. 头部压缩：
      1. 结构：。
      2. 静态字典：为高频出现在头部的字符串和字段建立了一张静态表，使用huffman编码压缩。
      3. 动态字典：为不在静态表范围内的头部字符串构建，在发送完某字段数据后，双方同步更新动态表，从index=62开始添加，这样在下次发送只需要发送1个字节的index号。
    2. 二进制帧：将响应报文划分为10种类型的帧，分为数据帧和控制帧，并采用二进制编码。
      1. 结构：
      2. 标志位携带简单的控制信息，如END_HEADERS表示头数据结束标志，END_Stream表示单方向数据发送结束，PRIORITY表示流的优先级。
      3. 流标识符标识该Frame属于哪个stream。
      4. 帧数据存放通过HPACK算法压缩过的HTTP头部和包体。
    3. 并发传输：多个Stream复用一条TCP连接达到并发。
      1. 结构：
      2. 一个TCP连接中有多个Stream，一个Stream里有多个message，相当于HTTP1中的请求或相应，由HTTP头部和包体构成，一条Message里包含多个Frame，它是HTTP2的最小单位，以二进制压缩格式存放HTTP1中的内容。
      3. 同一个HTTP请求与响应跑在一个Stream中，一个Frame由多个TCP报文构成。
      4. 不同帧可以乱序发送，由帧头部的流标识符区分并有序组装；同一Stream内部的帧必须严格有序。
      5. 可以根据控制位对每个Stream设置不同优先级。
      6. 双方都可以建立Stream，客户端建立的Stream必须是奇数号，而服务端必须是偶数号。
    4. 主动推送资源：客户端访问HTML时，服务器可以主动推送CSS文件。服务器通过PUSH_PROMISE帧传输HTTP头部，并通过Promised Stream ID字段告知客户端将会在哪个偶数号Stream中发送包体。
    5. 缺点：HTTP2基于TCP协议传递数据，TCP是字节流协议，必须保证收到的字节数据完整且连续，当前一个字节数据没有到达时，后收到的字节数据都会放在内核缓冲区，存在队头阻塞问题。
  10. HTTP3.0
    1. QUIC：基于UDP在应用层实现的协议。
      1. 无队头阻塞：Stream中的一个数据包丢失了，该流的其他数据包无法被HTTP3读取直到QUIC重传丢失的报文；其他流的数据报文可以正常被读取。
      2. 更快的连接建立：QUIC内部包含TLS，在自己的帧里会携带TLS里的记录；QUIC使用的是TLS1.3，因此仅需1个RTT就可以同时完成连接建立与密钥协商；第二次连接时应用数据包可以和QUIC握手信息一起发送，达到0RTT。
      3. 连接迁移：通过连接ID标记通信的两个端点，当移动设备网络变化时，只要仍保有上下文信息，就能无缝复用原连接，而不必像老版本一样再次握手以及慢启动。
    2. HTTP层
      1. 帧头部：直接使用QUIC里的Stream，帧头部只有类型和长度两个字段。
      2. 头部压缩算法：QPACK
        1. 静态表：扩大到９１项
        2. 动态表：如果首次出现的请求丢包，后续收到请求时无法解码出HPACK头部；QUIC使用Encoder Stream与Decoder Stream两个特殊的单向流同步双方的动态表。
  11. 远程过程调用RPC：本身并不是一种具体的协议，而是调用方式，例如在调用远程服务器的方法时屏蔽网络细节，像调用本地方法一样。
    1. HTTP主要用于b(rowser)/s(erver)架构，而RPC主要用于c(lient)/s(erver)架构。因为之前的软件大部分作为客户端只需要跟自家的服务端建立连接，因此可以使用自家造的RPC协议，而浏览器需要访问不同公司的服务器，所以需要统一的标准，即HTTP。
    2. 两者区别：
      1. 服务发现：找到服务对应的IP端口：RPC有专门的中间服务保存服务名和IP信息。
      2. 底层连接形式：RPC一般有连接池，复用连接。
      3. 传输消息：RPC采用体积更小的序列化协议保存结构体数据，因此性能比HTTP1.1更好一些。
  12. 长轮询：用于手机扫网页端二维码等场景。打开二维码时发送的HTTP请求增大超时延迟，在超时延迟内服务器只要收到扫码请求就立即返回给客户端网页，因此可以实现及时响应。
  13. WebSocket：实现全双工连接。
    1. 建立：在完成TCP握手与一次HTTP通信后，如果想建立WebSocket连接，就在HTTP header头中表示想升级成WebSocket协议，并附带一段随机生成的base64码。如果服务器支持升级，就用base64码用某个公开算法变成另一段字符串，同时带上101状态码（协议切换）返回给浏览器。浏览器使用同样的公开算法转换base64码来验证是否一致。
    2. 格式：
    3. opcode：数据类型
    4. payload长度：先读最开始的7位，如果是0-125表示payload全部长度；如果是126表示长度在126-65535之间，还会读取16bit；如果长度是127还会再读64bit。

3. TCP

  1. 三大特点：面向连接的（一对一）、可靠的（保证到达）、字节流（消息分组且有序）

  2. 第三次握手是可以携带数据的，而前两次不行。

  3. 必须三次握手的原因：

    1. 防止历史连接初始化连接：如果有之前申请建立连接的报文由于网络拥堵未能到达，客户端再次发送建立连接的请求，但是之前的报文先到达，这就叫历史连接。第三次握手可以让客户端确认连接是否为历史连接，从而在历史连接到达时终止连接。
    2. 同步双方初始序列号：双方发送的携带初始序列号的报文都需要得到应答，才能确保同步。四次握手的中间两次可以合并，而两次握手不能保证服务端的序列号得到应答。
    3. 避免资源浪费：历史连接建立会成为冗余的连接，造成资源浪费。

  4. TCP使用MSS而不用IP分片：IP层没有超时重传机制，如果一个IP分片丢失，整个IP报文的所有分片都得重传。

  5. SYN攻击：攻击者伪造不同IP发送SYN报文给服务端，占满服务端的半连接队列。

    1. 增大半连接队列。
    2. 改用cookie绕过半连接队列。
    3. 减少SYN+ACK重传次数从而加快半连接断开。

  6. 主动断开连接的一方才有TIME_WAIT状态。

    1. TIME_WAIT时间为2MSL（报文最大生存时间），从客户端接收到FIN后发送ACK开始计时，在接到重发的FIN后会重新开始计时。
    2. 为什么是2MSL：一般认为网络中可能还存在来自发送方的数据包，发送方接收到这些数据包的ACK时最长需要一来一回的时间。
    3. 为什么需要TIME_WAIT状态：防止历史连接的数据被后续连接接收；等待足够时间，以确保被动关闭连接的另一方能被正确关闭。
    4. 优化：
      1. net.ipv4.tcp_tw_reuse与tcp_timestamps：复用处于TIME_WAIT的socket为新的连接，还需要时间戳的支持以判断过期的历史连接数据。
      2. net.ipv4.tcp_max_tw_bucket：处于最大TIME_WAIT连接数时重置后面的TIME_WAIT连接状态。
      3. SO_LINGER：设置socket选项来设置调用close关闭连接行为，比如直接发送RST异常关闭。

  7. 建立连接后出现故障：

    1. 客户端：保活机制。
    2. 服务端：内核回收进程资源时会与客户端完成四次挥手。

  8. socket网络编程：

    1. 服务端接收到FIN报文后会在缓冲区后插入文件描述符EOF，应用程序在read到EOF时调用close关闭套接字，使服务器发出FIN包。

  9. 没有accept可以建立连接：accept只是从已建立的连接中取出一个可用的。

  10. 没有listen可以建立连接：客户端可以形成TCP自连接。

  11. 重传：

    1. 超时重传：动态计算RTO
    2. 快速重传：收到三个相同的ACK报文时重传丢失的报文段。
    3. SACK选择性确认：在TCP头部选项字段中加入SACK，将已收到的数据信息告知发送方。
    4. Duplicate SACK（D-SACK）：告知发送方哪些信息被重复接收。
      1. 让发送方知道使发出去的包丢了还是ACK丢了
      2. 知道是不是发送方的数据包被网络延迟
      3. 直到网络中是不是把发送方的数据包复制了

  12. 窗口：无需等待ACK而可以继续发送数据的最大值。接收方通过头部的Window字段告知接收窗口大小，发送方维护滑动窗口。

    1. 发送方的滑动窗口swnd：

      1. 结构：
      2. SND.WND：发送窗口的大小，由接收方指定
      3. SND.UNA：指向已发送但未确认的第一个字节的序列号
      4. SND.NXT：指向未发送但可发送的第一个字节序列号
      5. 可用窗口大小=SND.WND - (SND.NXT - SND.UNA)

    2. 接收方的滑动窗口rwnd

      1. 结构：
      2. RCV.WND：接收窗口大小，会告知发送方
      3. RCV.NXT：指向期望收到的下一个数据字节的序列号

    3. 接收窗口大小约等于发送窗口大小，但不完全相等。

  13. 流量控制：发送方根据接收方的实际接收能力控制发送的数据量。

    1. 应用程序读取能力不够时，会出现接收窗口收缩到0的死锁情况；如果操作系统直接减少接收缓冲区大小，甚至可能出现数据包丢失。TCP规定不允许同时减少缓存与收缩窗口，只能先收缩窗口。
    2. 窗口关闭：TCP一方收到另一方的零窗口通知时启动持续计时器，在超时时发送窗口探测报文。如果多次探测还是0则发送RST报文中断连接。
    3. 糊涂窗口综合症：TCP为了很少的数据发送报文。
      1. 接收方窗口大小小于min(MSS, 缓存空间/2)时发送零窗口通知
      2. （需要1满足）发送方使用Nagle算法，只有满足窗口大小>=MSS且数据大小>=MSS或收到之前发送数据的ACK时才发送数据，否则囤积数据。

  14. 拥塞控制：流量控制是避免发送方的数据填满接收方的缓冲区，而拥塞控制是避免发送方的数据填满整个网络。

    1. 拥塞窗口cwnd=min(swnd, rwnd)
    2. 慢启动：发送方每收到一个ACK，cwnd++。变化趋势：1->2->4->8->……，呈指数型增长。慢启动门限ssthresh（一般为65535）。当cwnd>=ssthresh时改为拥塞避免算法
    3. 拥塞避免：每收到一个ACK，cwnd+=1/cwnd。变成线性增长。网络慢慢进入拥塞，最终在出现丢包需要进行重传时进入拥塞发生。
    4. 拥塞发生：
      1. 拥塞发生：触发超时重传时使用。ssthresh=cwnd/2，cwnd重置为初始值，重新开始慢启动。
      2. 快速重传：触发快速重传时使用，是对拥塞发生的优化。cwnd/=2，ssthresh=cwnd，进入快速恢复算法。
    5. 快速回复：认为能连续收到3个重复ACK时网络状况并不那么糟糕，所以不需要像超时重传那样反应剧烈。
      1. cwnd+=3（在快速重传的基础上），表示确认3个数据包被收到。
      2. 重传丢失的数据包
      3. 如果再收到重复的ACK，cwnd++。
      4. 如果收到新数据的ACK，将cwnd设置为慢启动的ssthresh值（ACK确认新数据，说明重复ACK的数据都已经收到，恢复过程已经结束），进入拥塞避免状态。

  15. TCP挥手有时候是3次的原因：被动断开方没有数据要发送，且开启了TCP延迟确认机制，第二次与第三次挥手会合并。

  16. 第三次握手如果丢失：

    1. 服务端一直重传SYN+ACK包，直到最大重传次数后断开连接。
    2. 客户端如果没有发送数据，一直处于ESTABLISHED状态，然后经过很长一段时间后发现一个死亡连接，于是断开；如果发送数据，那么一直收不到ACK就会一直重传到最大次数，然后断开连接。

  17. TCP Fast Open

    1. 第一次建立连接时，服务端在第二次握手时产生一个加密Cookie与SYN+ACK包一起发送，客户端在第三次握手时将ACK与HTTP请求一起发送。
    2. 下次请求时客户端在SYN包中带上Cookie，服务端能直接从Cookie获取TCP相关信息，跳过握手并返回SYN+ACK+数据，只需要一个RTT。
    3. 需要双方同时支持。

  18. TCP延迟确认

    1. 有响应数据发送时，ACK随响应数据一起发送
    2. 没有响应数据发送时，ACK等待一段时间
    3. 在等待期间对方第二个数据报文到达，则ACK立刻发送
    4. 与Nagle算法冲突，一方没有发送数据等待下一个报文到达，一方等待当前报文的确认，会造成额外时延

  19. TCP全连接队列已满时，是否需要放弃连接：通过修改tcp_abort_on_overflow参数。

    1. 0：如果队列已满，server扔掉client发过来的ack。
    2. 1：如果队列已满，server发送RST包给client，表示废除握手过程和连接。
    3. 一般应当设置为0，因为有时服务器只是短暂繁忙，当队列有空位时再次接收到ACK请求仍然能成功建立连接。

  20. close与shutdown：

    1. close：完全断开连接，无法发送数据与传输数据，调用close一方的连接叫孤儿连接，进程名为空。
    2. shutdown：
      1. SHUT_RD(0)：关闭连接的读方向。立即丢弃接收缓冲区的数据，并在之后有数据到来时返回ACK并丢弃。
      2. SHUT_WR(1)：关闭连接的写方向。立即发送发送缓冲区未发送的数据，并发送FIN报文。
      3. SHUT_RDWR(2)：1和2各执行一次。

  21. TCP连接双方同时关闭：两方发送FIN报文后等待ACK，都等来了FIN报文，因此都进入CLOSING状态，之后双方回复ACK确认对方发送通道关闭，进入TIME_WAIT状态。

  22. 如何理解TCP是面向字节流的协议，而UDP是面向报文的协议

    1. 用户协议通过UDP协议传输时，操作系统不会对消息进行拆分，这意味着每个UDP报文代表一条完整的用户消息。
    2. 用户协议通过TCP协议传输时，消息可能会被分成多个TCP报文，而拆分方式不固定，不能认为一个用户消息对应一个TCP报文。（TCP协议传输的数据粒度比报文更细）
    3. 解决粘包：固定长度；特殊字符作为边界（如HTTP以回车+换行作为边界）；自定义消息结构（包头固定大小，在包头中记录数据大小）。

  23. TCP时间戳：

    1. 精确计算RTT
    2. 防止序列号回绕（随机出相同的SYN）：判断新收到的数据包的时间戳是否在递增序列中，如果不在表示该数据包过期。

  24. 处于ESTABLISH状态的服务端收到了客户端的SYN报文：

    1. 端口号与历史连接不同：服务端认为需要建立新的连接。旧连接如果发送了数据，会返回RST报文强制关闭，否则在TCP保活机制下释放。
    2. 端口号与历史连接相同：服务端会回复一个携带了正确序列号和确认号的ACK（Challenge ACK）。客户端收到后发现序列号不是自己所期望的，就会返回RST报文。

  25. 挥手时被动关闭端的FIN报文比发送的数据更早到达另一方：FIN报文被当作乱序报文储存起来，等按顺序收到并处理完数据报文后，处理FIN报文再进入TIME_WAIT状态。
