学习笔记-docker与k8s
发表于：2023-05-21 | 分类： 学习笔记

1. docker

1、与虚拟机的区别

  • 虚拟化技术依赖物理CPU和内存，是硬件级别的；Docker构建在操作系统层面，利用操作系统的容器化技术。
  • Docker较为轻量级，部署的应用完全隔离。
  • 传统虚拟化技术通过快照保存状态；Docker使用类似源码管理的机制，保存容器的快照历史版本并切换，成本低。
  • 传统虚拟化技术构建系统复杂；Docker可通过Dockerfile文件构建容器，且文件可手动编写，利于持续交付。

优势：

  • 灵活：可装箱复杂应用
  • 轻量级：共享主机内核
  • 可互换：即时部署、更新和升级
  • 便携式：本地构建、部署到云、任何地方运行

好记版：

  • 产品：产品交付方便
  • 开发：简化环境配置
  • 测试：多版本测试
  • 运维：环境一致性
  • 架构：自动化扩容

2、架构

大致是C/S架构
Client端：

  • 命令行CLI接口，与REST API Server交互（用户使用的Docker命令）
  • REST API Server，与Dockerd交互

后端：

  • Docker Daemon
    • Docker Server：C/S的服务端，接受并调度分发Client发送的请求（路由），找到相应的Handler执行。
    • Engine：运行引擎，存储container并通过执行job来管理它们。通过handler对象来执行特定job。
    • Job：Engine的工作单元，所有工作的抽象表示。类似于Unix的进程。
  • Docker Registry：注册中心，作为云端镜像仓库使用。
  • Graph：存储模块
    • Repository：保管已下载的镜像。同一repository内镜像为同一类，通过tag区分版本。
    • GraphDB：记录已下载镜像关系。
  • Driver：执行模块，实现对容器执行环境的定制。
    • graphdriver：管理镜像，包括存储与获取。即与graph交互。
    • networkdriver：配置容器网络环境。docker启动时创建网桥、专属虚拟网卡、分配IP、端口并与宿主机做端口映射、设置容器防火墙。
    • execdriver：创建容器运行namespace，统计、限制资源使用，真正运行容器内部进程。
  • libcontainer：函数库，封装接口满足上层对容器管理的需求。希望该库可以不依靠任何依赖，直接访问内核中容器相关API。Docker通过调用库操纵容器的namespace、cgroups、网络设备等。
  • container：服务交付的最终体现形式。

容器层记录对镜像的修改，所有镜像层都是只读的。镜像可以被多个容器共享。

3、Dockerfile

用于构建镜像的脚本

1、FROM

必须是第一条指令，指定基础镜像。
特殊镜像scratch：虚拟概念，表示一个空白的镜像

2、RUN

执行命令行命令，有两种格式。

  • shell格式：RUN <命令>，像直接在命令行中输入命名一样。
  • exec格式：RUN [“可执行文件”, “参数1”, “参数2”]。

区别：exec格式可以接受参数，而shell格式忽略参数。shell格式相当于在前面还要添加/bin/sh -c，所以app启动进程ID不是1。
Dockerfile中每个指令都会建立一层，在其上执行命令，然后commit该层修改，构成新的镜像。所以应该避免创建过多层。

3、操作

  • create：初建状态
    • 为只读文件系统添加一层可读写层，即静态容器
  • start：运行状态
    • 为可读写文件系统添加一个进程空间，生成动态容器
  • run：create + start
  • stop：停止状态
    • 发送SIGTERM信号，停止所有进程。清理完进程后再SIGKILL退出
  • kill：停止状态
    • 发送SIGKILL，快速强制关闭
  • pause：暂停状态
    • 暂停所有进程
  • unpause：运行状态
  • delete：删除
  • commit：
    • 把容器的可读写层转化为只读层
  • build：run + 进程修改数据 + commit。不断循环直到生成所需镜像

4、数据共享与持久化

1、数据卷

一个可供一个或多个容器使用的特殊目录，绕过UFS，主要特性包括：

  • 可以在容器间共享和重用
  • 修改立马生效
  • 更新不会影响镜像
  • 默认一直存在，即使容器被删除

2、挂载主机目录

可以挂在一个本地主机的目录到容器中去。

5、网络模式

1、Bridge模式

Docker进程启动时，会在主机上创建名为docker0的虚拟网桥，此主机上启动的Docker容器会连接到该虚拟网桥上。
虚拟网桥工作方式和物理交换机类似，主机上所有容器就通过交换机连接在了一个二层网络中。
从docker0子网中分配一个IP给容器用，并设置docker0的IP地址为容器默认网关。
在主机上创建一对虚拟网卡veth pair设备，Docker将设备一端放在新创建的容器中，并命名为eth0（容器网卡），另一端放在主机中，命名类似vethxxx，并将该网络设备加入docker0网桥中。

2、Host模式

容器不会获得独立的Network Namespace，而是和宿主机共用一个。容器使用宿主机的IP和端口，但是其他方面，如文件系统、进程列表还是和宿主机隔离。

3、Container

指定新创建的容器和已存在的容器共享Network Namespace。两个容器可以通过网卡通信。

4、None

Docker容器拥有自己的Network Namespace，但是并不添加任何网络配置。

6、实现原理

资源隔离：借助Linux内核的Namespace技术实现，通过在新的Namespace下通过mount重新挂载对应的文件夹（Linux的一切皆文件思想）来屏蔽父进程信息。

  • 进程隔离：一个docker相当于宿主机的一个进程。在子进程内部重新挂载proc目录（存储当前内核运行状态的一系列特殊文件），屏蔽父进程的进程信息。
  • 文件隔离：将根目录挂载到了另一个目录。

资源限制：通过Linux内核的cgroups（control groups）机制实现，对cpu、内存等资源实现精细化控制。

  • api以伪文件系统的实现方式，用户程序可通过文件系统实现cgroup组件管理。
  • 组件管理操作单元可以细粒度到线程级别，用户可创建/销毁cgroup来实现资源再分配。
  • 所有资源管理功能都以子系统方式实现。

cgroup功能：

  • 资源限制：控制任务使用的资源总额。
  • 优先级分配：通过分配cpu时间片数量与磁盘IO带宽，控制任务运行优先级。
  • 资源统计：统计系统的资源使用量。
  • 任务控制：挂起、恢复任务。

2. kubernetes

提供集中式管理集群机器、应用，实现命令管理集群增减、版本变更，不停机灰度更新，确保高可用、高性能、高扩展。

  • 服务发现和负载均衡：使用DNS名称或自己的IP地址来暴露容器。负载均衡并分配网络流量，从而稳定部署。
  • 存储编排：允许自动挂载选择的存储系统。
  • 自动部署和回滚：可以将已部署容器的实际状态更改为用户描述的期望状态。
  • 自动完成装箱计算：告诉K8s每个容器需要的CPU和内存，K8s可以将这些容器按实际情况调度到节点上，最佳利用资源。
  • 自我修复：重启失败容器、替换容器、杀死不响应用户定义的运行状态检查的容器。
  • 密钥与配置管理：允许存储和管理敏感信息，可在不重建容器镜像的情况下部署和更新密钥和应用程序配置。

1、集群架构

1、概念

  • master：主节点，控制平台，通常一个
    • API Server：K8s所有组件都依赖其进行消息传送。
    • Controller：管理集群状态，如自动修复、水平扩张。
    • Scheduler：调度器，自动寻找合适的Pod防止container。
    • etcd：分布式存储系统，存储API Server的原信息。
  • worker：工作节点，运行具体任务，由主节点管理
    • Pod
    • kubelet：通过API Server接收到所需要Pod运行的状态，提交到Container Runtime组件中。
    • Storage Plugin：配置存储。  
    • Network Plugin：配置网络。  
    • kube-proxy：利用iptable的能力组件k8s的network，即cluster network。
  • Pod：K8s调度、管理的最小单位，一个主节点控制多个Node，一个Node运行多个Pod，一个Pod可包含多个容器，主节点考量负载自动调度pod到哪个节点运行。

2、架构

  • 核心层：对外提供API构建高层应用，对内提供插件式应用执行环境
  • 应用层：部署和路由（服务发现、DNS解析等）
  • 管理层：系统度量（基础设施、容器和网络度量）、自动化、策略管理
  • 接口层：kubectl命令行工具、客户端SDK、集群联邦
  • 生态系统：
    • K8s外部：日志、监控、配置管理等  
    • K8s内部：    
      • CRI：容器运行时接口，定义了容器和镜像的服务接口    
      • CNI：容器网络接口，定义了Linux容器的网络接口规范和库    
      • CVI：容器存储接口：使容器编排系统可将任意存储系统暴露给自己的容器工作负载

3、大致流程

用户提交Pod给K8s部署

  • Pod请求提交给API Server
  • API Server将信息写入etcd
  • Scheduler通过API Server的notification机制得到调度信息
  • Scheduler根据内存状态调度，然后通知API Server调度的节点
  • API Server将调度信息写入etcd中，通知相应的节点启动Pod
  • 节点的kubelet收到通知
    • 调Container runtime启动配置容器和容器的运行环境
    • 调Storage Plugin配置存储
    • 调Network Plugin配置网络

2、核心概念

1、资源

组成结构与yaml文件描述的对应部分pod的内容都是通过json/yaml文件表达的，描述该pod资源的组成部分。

  • apiVersion：描述正在操作的资源
  • kind：资源类型，如pod、deployment等
  • metadata：元信息，如创建时间、标签、名字、namespace等。  
    • labels：资源标签，为key：value型元数据，主要用于筛选资源和组合资源  
    • annotations：存储资源的非标示性信息，用于扩展资源的spec/status描述
  • spec：希望资源达到的预期状态，如内部需要运行的container、storage、volume等  
    • replicas：期望的pod数目，默认1  
    • selector：可选字段，指定deployment管理的pod范围，必须匹配spec.template.metadata.labels  
    • template：跟pod的yaml文件格式一样，不需要apiVersion、kind字段，指示了最终创建出来的Pod采用的Pod.spec  
    • strategy：更新策略，默认RollingUpdate
  • status：资源当前观测到的状态，如pod可能正被调度、running等

2、Pod

容器是单进程模型，Pod可类比为进程组管理容器。Pod是K8s集群中运行部署应用或服务的最小单元。

  • 同一个Pod内为超亲密关系、紧密协作 的容器，无法单独调度。  
    • 进程间文件交换，如读写日志  
    • 进程间通过localhost通信  
    • 进程间RPC调用频繁  
    • 进程间共享linux namespace支持多个容器在一个Pod中共享网络地址、文件系统。

3、controller manager

通过API Server监控整个集群的状态，并确保集群处于预期工作状态。

4、ReplicaSet

适合无状态服务部署。

用户创建指定数量pod，确保数量符合预期，且支持滚动式自动扩容缩容。

很少单独使用，一般用于deployment中，被其使用。

5、Deployment

适合无状态服务部署。

一个deployment对应多个pod，表示用户对K8s集群的一次更新操作，如创建一个新服务、更新一个新服务、滚动升级一个服务等。

在Depolyment中描述想要的目标状态，controller就会将pod和RS的实际状态改变到目标状态。

工作在ReplicaSet上，用于管理tomcat、nginx等无状态应用。

Deplooyment只负责管理不同版本的ReplicaSet，由ReplicaSet管理具体Pod书，每个ReplicaSet对应Deployment template的一个版本。

版本更新方式：

  • 蓝绿部署：不停老版本，部署新版本测试，测试完成后将流量切到新版本，同时老版本升级。无需停机，风险小。
  • 滚动发布：服务器轮流停机，更新后再投入使用。相对于蓝绿部署更节约资源。
  • 灰度发布：让一部分用户开始用新版本，如果没有明显问题再逐步扩大范围，迁移用户。  
    • 金丝雀发布
    • 版本回退：更改spec中的template部分。默认情况下k8s保存前两次deployment的历史记录，可以随时回退。扩容缩容时更改的是spec中的replicas数目，因此不会记录。

6、DaemonSet

后台支撑服务集（无状态）。

存储、日志、监控等在每个节点上支持K8s集群运行的服务。保证每个Node都有一个此类Pod运行。

7、StatefulSet

适合有状态服务部署。

无状态服务中，控制的Pod名字随机设置，出故障则被丢弃，在另一个地方重启一个新pod。重要的只是Pod总数。

StatefulSet控制有状态服务，每个Pod名字事先确定不能更改，关联与该Pod对应的状态。重启Pod时需要挂载上原Pod的存储继续其状态。

StatefulSet直接管理下属Pod，而不使用ReplicaSet。

适合其的业务包括数据库服务、集群化管理服务等有状态服务，如mysql。、

特性：

  • 每个Pod都有稳定、唯一的网络标识
  • Pod副本启停顺序受控
  • Pod采用稳定的持久化存储

控制流程：

  • 注册Informer的Event Handler，处理StatefulSet和Pod的变化
  • StatefulSet Controller中，每次收到StatefulSet/Pod的变化，都会找到对应的StatefulSet放到队列。
  • 从队列中取出处理时，先查看拿到的StatefulSet的template有无对应的Controller Revision，没有则说明更新过，就创建一个新版本的Revision。
  • Controller拿出所有版本号，按序号整理一遍，并创建/删除冲突的Pod。
  • Controller查看是否需要更新Pod。

8、Job

定时执行任务（无状态）用来控制批处理型任务的API对象。

Job管理的Pod根据用户的设置，成功完成任务后自动退出。可通过参数设置失败时重启次数、并行数等。

Job一次性，Cronjob用于周期性执行任务。

9、Volume-存储卷

管理K8s存储，声明在Pod的容器可以访问文件目录。

  • 容器异常退出恢复时恢复数据
  • Pod内多个容器共享数据

本身是抽象概念，可以支持多种后端存储。生命周期和作用范围为一个Pod。

阶段：

  • create：创建PVC对象、真正的存储PV对象并绑定
  • attach：将存储挂载到node上
  • mount：将存储挂载到node可使用的路径上

1、StorageClass

存储类PV的配置文件。

2、PV（Persistent Volume）-持久存储卷

用于解耦Pod和Volume之间生命周期的关联，使得Pod删除后，其使用的PV仍然存在。

产生方式：

  • 静态Provisioning：集群管理员预创建一些PV，用户提交PVC时找到合适的PV绑定。
  • 动态Provisioning：集群管理员编写模板文件，用户提交PVC时指定使用的存储模板（StorageClass），K8s根据这些动态生成PV并绑定。

Yaml的spec中可规定：存储大小、访问模式（单/多node读写/只读访问）、PVC删除后PV的处理等。

状态流转：

  • 创建PV后短暂pending
  • available
  • 绑定PVC后bound
  • PVC删除后released：此时没办法直接变回available  
    • 将该PV对象信息填到一个新建的PV对象  
    • 使用完后不删除PVC

3、PVC（Persistent Volume Claim）

用户通过PVC使用PV，在PVC中声明所需的存储大小以及访问模式，PV由集群管理员与存储团队统一管控。

类似于类的封装。

10、Service-服务

作用：

  • 负载均衡：一个deployment可能有多个相同Pod，在外部访问时做一次负载均衡。
  • 固定外部IP：不同Pod有不同IP、Pod重启时IP和端口号会变化，因此不能以确定的IP和端口号提供服务。  
    • ervice相当于把Pod的可变IP映射到一个固定IP供外部访问

11、Federation-集群联邦

提供跨Region跨服务商的K8s集群服务。

每个联邦有自己的分布式存储、API Server、Controller Manager。

12、Namespace-命名空间

集群内部隔离，K8s每个资源都属于一个Namespace。

初始默认命名空间default和系统命名空间kube-system。

一般可根据项目属性，如生产、测试、开发划分。

3、声明式API & 控制器模式

控制器模式的核心为控制循环概念，包括三个组件：控制器、被控制系统、观测系统的传感器。

大概流程：

  • 外界修改资源spec
  • 控制器比较spec与status，计算diff
  • diff被用来决定执行对系统进行怎样的控制操作
  • 控制操作使得系统产生新输出，传感器以status形式上报
  • 系统不断向spec趋近

命令式API告诉系统要执行的操作，而声明式API只告诉系统最终需要达到的状态，不管系统具体执行的操作。

优劣：

  • 错误处理：命令式API错误时重试可能导致系统执行多次、执行命令错误等问题。
  • 巡检系统：为了处理命令超时导致的数据不一致问题，往往会设置一个巡检系统。
  • 并发访问：多方并发操作资源时，难以保证操作生效情况，而加锁又降低效率。

4、网络

1、约束

  • 任意两个pod可直接通信
  • node和pod可直接通信
  • pod看到自己的IP跟别人看见它的应该一致

2、Network Namespace

实现网络虚拟化的基础，创建了隔离的网络空间。

每个pod有独立的网络空间，pod net container共享这个网络空间，K8s通过接口在pod net container之间通信。

所有的container通过pod的ip对外提供服务。

3、Service

集群网络存在的问题：

  • Pod容易重启，IP地址会发生变化，不能用固定IP地址访问
  • Pod组需要提供统一的访问入口、进行负载均衡

通过service实现。

Service的yaml文件中，通过spec.selector选择目标pod，通过spec.ports字段指定协议、端口以及目标端口等信息。

创建完毕后，Service就会对Pod组外部提供一个虚拟IP，集群中其他Pod可以通过该虚拟IP访问Service。

Headless Service创建时可以告诉K8s不需要clusterIP-虚拟IP，访问时直接通过service_name用DNS方式解析到所有后端的IP地址，由客户端自己选择一个IP地址访问。向外暴露Service

  • NodePort：在集群Node上暴露节点的一个端口，相当于多一次转发
  • LoadBalancer：在NodePort上再做一次转发，即在所有节点前又挂一个SLB

5、调度

1、资源调度

填在pod.spec.containers.resources里。

  • request：对pod基本保底的资源要求
  • limits：对pod可用能力上限的限制

四大类资源：

  • CPU
  • memory
  • 临时存储
  • 通用扩展资源，如GPU

2、关系调度

  • Pod亲和调度：PodAffinity
  • Pod反亲和调度：PodAntAffinity

有两种策略：

  • required：必须这样，否则调度失败
  • preferred：优先调度

3、优先级调度

创建priorityClass，再为每个Pod配置不同的priorityClassName。

6、容器设计模式：sidecar

在Pod里定义一些专门的容器，执行主业务容器所需要的一些辅助工作。

只是设计模式，不是k8s官方提出的概念。

特点：

  • 运行环境和编程语言与主程序独立。
  • 资源访问权限与主程序相同。
  • 靠近主程序，所以通信没有明显延迟。
  • 通过作为自己的进程附加到住应用所在的主机/子容器中，可扩展不提供扩展机制的程序。

适用场合/优势：

  • 与主程序使用不同框架/语言开发
  • 抽象功能到不同层，降低微服务代码复杂性，降低耦合度，提升重用度
  • 独立升级更新

如：

  • 日志：Volume在Pod里共享，业务容器将日志写在Volume里，日志容器再共享读日志，存到remote write中或转发。
  • 代理容器：Pod中是一个集群，通过proxy容器访问外部服务。
  • 适配器：Adapter容器在系统内外传输消息时做格式转换。

不适用：

  • 需要优化进程通信。
  • 小型应用，sidecar部署资源成本可能超过应用本身。
  • 服务需要独立伸缩。

所有设计模式的本质都是：解耦和重用。

与DaemonSet方式的对比

  • 部署成本：Sidecar对每个Pod都需要部署，DaemonSet只需要部署并维护DaemonSet
  • 定制化：Sidecar可对每个Pod单独配置，DaemonSet受限，可通过容器/路径等映射
  • 多租户隔离：Sidecar通过容器隔离，可单独分配资源，DaemonSet只能通过配置隔离
  • 支持集群规模：Sidecar无限制，DaemonSet只支持中小型规模
  • 资源占用：Sidecar每个Pod运行一个容器，较高，DaemonSet每个节点运行一个容器，较低

总结：Sidecar适合大型、混合型的PaaS集群，隔离型、灵活性好；而DaemonSet适合功能较单一（部门内部/产品自建）的集群，资源占用低。
