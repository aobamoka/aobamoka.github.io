学习笔记-分布式系统
发表于：2023-01-28 | 分类： 学习笔记

1. 基本概念

1. CAP定理

对于一个分布式系统，以下三点不能同时很好的满足。

  • 一致性Consistency：所有节点在同一物理时间具有相同数据。每次请求要么获得最新的数据，要么失败。强调数据正确。
  • 可用性Availability：保证每个请求不管成功或失败都有响应。（不保证是最新数据）强调不出错。
  • 分区容错性Partition tolerance：任意节点故障时系统仍可以对外服务。强调不挂。

排列组合可得：

  • CA：单点集群，可扩展性不强。
  • CP：性能不高，因为需要一段时间同步，这段时间没法回复请求。
  • AP：对一致性要求较低。有些节点可能请求到较旧的数据。

BASE：NoSQL数据库对可用性及一致性的弱要求原则：

  • 基本可用Basically Available
  • 软状态/柔性事务Soft-state：无连接
  • 最终一致性Eventually Consistency：ACID的最终目的

1. 数据一致性

由于代价过高，一般用弱一致性提高性能

  • 强一致性：无论更新在哪一个副本执行，之后所有读操作都要获得最新数据。
  • 弱一致性：读到最新数据需要一段时间，称为不一致性窗口。
  • 最终一致性：弱一致性的特例，保证最终（尽量长）能读到更新。

解决方案

  • 分布式事务：两段提交
  • 分布式锁
  • 消息队列、消息持久化、重试、幂等操作
  • Raft等一致性算法

2. 服务可用性

只要收到用户的请求，服务器就必须给出相应。

解决方案：

  • 负载均衡
  • 降级：根据业务情况及流量对一些服务和页面有策略的降级，以释放服务器资源、保证核心用户正常运行。
  • 熔断：对于大量超时或失败时应熔断该服务所有调用，直到服务好转。
  • 流量控制
  • 异地多活：不同地区维护不同子系统。

3. 分区容错性

对于分布式系统而言，分区容错性是前提，必须保证。因而数据同步的时延势必导致C和A不能同时满足。

2. 数据一致性

另：事务一致性指的就是ACID

1. 基本要求

  1. 可终止性：一致的结果在有限时间内能完成。
  2. 共识性：不同节点最终完成决策的结果应该相同。
  3. 合法性：决策的结果必须是其他进程提出的提案。

2. 强一致性

系统中某个数据被更新后，后续的任何读取都应该读到更新后的值。

约束强度：线性 > 顺序 > 因果

1. 线性一致性

原子一致性、严格一致性。

比顺序一致性多的条件：如果不同进程的事件不是并发时间，那么重排后的顺序与时钟顺序一致。

2. 顺序一致性

条件：

  1. 同一进程内的事件先后顺序不能改变。
  2. 对同一变量的读写，每次读到的都是最近写入的数据。（不允许先读出新值再读出旧值）

对不同进程的事件之间的顺序没有要求，即使它们在时钟上有早晚。

3. 因果一致性

条件：

  1. 满足因果关系的事件必须按因果关系先后出现。（比如：评论与回复）
  2. 同一进程内的事件先后顺序不能改变。

3. 带约束的一致性

强一致性系统较难实现，适当放宽成为弱一致性，即在某个时刻系统达到一致。

最终一致性/乐观复制：保证能达到一致性，但不能保证达到的时间（窗口时间）。

4. 一致性与共识性

一致性描述系统一组进程在某个时刻的状态，是系统静态状态；共识描述一组进程间协作的动态过程，目的是在相关进程间确定下一个操作。

2. 高并发系统设计

1. 拆分子系统

2. 缓存

利用大部分高并发读多写少的特点。

3. 消息队列

利用异步写的消息队列。

4. 分库分表

5. 读写分离

主从架构，主表写从表读。

6. 高并发服务器方案

  • 应用程序与静态资源分离，将静态资源（图片、视频、js、css等）单独保存到专门的静态资源服务器
  • 客户端缓存
  • 反向代理与负载均衡：请求先经过反向代理服务器，它可以直接返回结果，也可以根据后端服务器的负载，将请求交给其中一台处理。感觉前三点Nginx都能做到。
  • 集群和分布式。集群指所有服务器都有相同的功能，起分流作用；分布式将不同的业务放到不同服务器中，请求一个业务可能需要使用多台服务器，加快处理速度。它们的作用都是将原属于一台服务器的压力分散到多台，同时加快请求速度

3. 分布式缓存

1. 应用场景

  • 页面缓存：缓存HTML、CSS和图片等Web页面内容。
  • 并行处理：中间计算结果需要共享。

2. 缓存雪崩/击穿

原有缓存失效而新缓存未到，导致所有应该访问缓存的请求都去访问数据库，造成巨大压力。

缓存雪崩是大规模的key失效，而缓存击穿是大量并发集中访问某一个key，而这个key失效。

一般多个缓存采用相同过期时间，同一时刻大量缓存过期。

解决：

  • 热点数据设置永不过期；其他数据合理分配过期时间。
  • 竞争互斥锁查询数据库，但会导致系统性能变差。

3. 缓存穿透

数据库中没有的数据，用户查缓存一遍找不到（缓存不会缓存空结果），还要去数据块中再查一遍，相当于两次无用查询。

解决：

  • 缓存空对象
  • 布隆过滤器：有变量被加入集合时，通过K个映射函数映射成位图中K个点并置1，查询时先查这些点是不是1，是则可能存在（可能有哈希碰撞），不是一定不存在。

4. 缓存预热

预加载。

5. 缓存淘汰

定时清理过期缓存：维护大量缓存key较为麻烦。

用户请求过来再判断是否过期，过期则去更新：每次用户过来都要判断，逻辑复杂。

6. 缓存降级

访问量剧增、服务出现问题或非核心服务影响到核心服务性能时，仍需要保证服务可用，即使有损服务。

目的是保证核心服务可用，即使有损。

解决前需要先梳理哪些服务可降级，哪些不行。

解决：数据库出现问题时，可以不去查询而直接返回默认值。

4. 分布式锁

redis和zookeeper都还没看过，先占个坑

1. Redis Redlock算法

客户端和多个独立的Redis节点依次请求加锁，如果客户端和半数以上的节点成功加锁，则认为客户端获得锁。

过程：

  • 客户端获取时间t1。
  • 客户端按顺序依次向N个Redis节点执行加锁操作：
    • 使用SET，带上NX、EX/PX与客户端唯一标识。
    • 为了保证Redis节点故障时算法运行，给加锁操作设置过期时间，远小于锁的过期时间。
  • 一旦客户端从超过半数节点上获取到锁，再次获取时间t2，计算消耗的时间，如果小于锁的过期时间，认为加锁成功。

加锁成功需要满足：

  • 客户端从超过半数节点获取到锁。
  • 获取锁的时间消耗小于锁的过期时间。

加锁成功后，锁的过期时间减去消耗时间即为有效时间。如果有效时间来不及完成共享数据，可以先释放锁。

释放锁：客户端向所有节点发起释放锁的操作。

5. 分布式事务

1. 2PC/XA方案

两阶段提交。每个数据库都回复ok，事务管理器再提交事务，并在各个数据库上操作，否则回滚。

适用单块应用里，跨多个库的分布式事务。严重依赖于数据库层面搞定复杂事务，效率很低，不适合高并发。

2. TCC强一致性方案

  • Try：对各个服务的资源做检测，然后锁定或预留资源。
  • Confirm：在各个服务中执行实际操作。
  • Cancel：如果任何一个服务方法执行出错，则将已成功的服务回滚。

事务回滚严重依赖于自己写逻辑实现回滚和补偿，造成巨大补偿代码量。

3. 可靠消息最终一致性方案

基于MQ（消息队列）实现事务。

  • A系统发送prepared消息到MQ，如果发送失败就取消操作，成功则进行下一步。
  • 执行本地事务，如果成功就告诉MQ发送确认消息，失败则告诉MQ回滚消息。
  • 如果MQ没收到确认消息，就自动定时轮询发送prepared消息的A系统，问本地事务是否处理成功。
  • A检查本地事务状态，确认是否成功并告知MQ。
  • 如果发送了确认消息，B系统接到确认消息，执行本地事务。
  • 如果B系统事务失败，就重试，或想办法通知A系统失败回滚。

4. 最大努力通知方案

  • A系统本地事务执行完后发消息到MQ。
  • 有专门消费MQ的最大努力通知服务，消费MQ并记录，然后通知B系统。
  • B系统执行成功则结束；否则最大努力通知服务定时尝试重新调用系统，达到指定次数后放弃。

6. 消息队列

MQ Message Queue，一般作为单独的中间件产品存在，独立部署。

1. 作用

  • 解耦：将调用关系/生产消费关系用中间件隔离，生产者只需要向MQ写消息，消费者只需要从MQ读消息，双方不必关心对方成员的增减、是否消费成功。
  • 异步：不需要将数据一个个同步发送过去再响应成功，而是直接将数据写到MQ里即可响应成功。
  • 削峰：将某一段时间的超高流量分摊到一段长时间内消化，防止被流量洪峰击垮系统。

2. 缺陷

  • 增加了系统复杂性：需要考虑消息是否丢失、消息积压、MQ满、消息重复等问题。
  • 降低系统可用性：各个系统强依赖MQ，需要额外保证MQ高可用。
  • 一致性问题：部分系统消费成功而其它系统消费失败。

7. 微服务

一种架构风格。将一个项目拆成独立的多个服务，每个服务占用线程，独立运行。

本质是有效拆分应用，实现便捷开发和部署。

1. 为什么

传统IT行业，软件大多是独立系统的堆砌。扩展性差、可靠性不高、维护成本高。

SOA服务化，但是其早期使用了总线模式，与某种技术栈强绑定，导致很多企业的遗留系统很难对接，切换时间长，成本高，新系统稳定性的收敛也需要时间。

2. 与单体架构的区别

  • 耦合性：
    • 单体架构所有模块耦合在一起，代码量大、维护困难
    • 微服务每个模块相当于一个独立项目，代码量小、遇到问题解决容易
  • 存储方式：
    • 单体架构所有模块共用一个数据库，存储方式单一
    • 微服务每个模块存储方式不同，数据库也是独立使用
  • 开发技术：
    • 单体架构所有模块开发技术一致
    • 微服务每个模块都可以使用不同开发技术，开发模式更灵活

3. 特点

  • 解耦：同一系统内服务大部分可解耦，因此应用可被轻易构建、修改和升级
  • 组件化：相互独立的组件，可轻易替换升级
  • 职责明确：每个微服务可专注于某种单一能力
  • 自治：开发者和团队可独立工作，提高开发速度
  • 持续交付：允许持续发布新版本，通过系统化自动手段来创建、测试和批准新版本
  • 去中心化管理：没有标准化的计数模式，开发者可选择最好的工具解决自己的问题
  • 敏捷性：支持敏捷开发，任何新功能都可被快速开发或丢弃

4. 优势

  • 独立开发
  • 独立部署
  • 错误隔离：某个服务故障不影响整个系统
  • 混合技术栈：可使用不同语言和技术为同一个应用构建不同服务
  • 按粒度扩展：可根据需求扩展某一个组件
