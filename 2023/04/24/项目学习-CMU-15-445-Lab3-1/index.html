项目学习-CMU-15-445-Lab3-1
发表于：2023-04-24 | 分类： 项目学习

1. Bustub架构

Lab1写的是实现缓存池替换的buffer_pool_manager，Lab2写的是B+树的底层实现，都是整个数据库的一个局部。

Lab3需要实现具体的数据库语句逻辑，所以需要先了解数据库整体的架构。

1. Parser

sql语句通过Parser生成一棵AST抽象语法树。

一般不会成为性能瓶颈，所以采用第三方库生成。

bustub采用了libpg_query库生成AST。

2. Binder

Binder将AST中的词语绑定到数据库实体上。

比如，对于sql语句

select与from是关键字，colA与table1是标识符。Binder遍历AST，将这些词语绑定到相应的实体，在bustub中属于各种类。

最终得到一棵bustub能理解的树，称为bustub AST。

3. Planner

Planner遍历bustub AST，得到初步的查询计划。查询计划规定了数据的流向，数据从树叶流向树根，在根节点输出结果。

对于sql语句

生成的未经优化的查询计划为：

  • 需要查t1、t2两个表，对应的DataSrouce（在bustub里为Scan）负责将数据捞上来。
  • Join算子，将两个表按 t1.c = t2.c连接。
  • Selection过滤出 t1.a > 5的数据。
  • Projection将b列投影出来。

本项目中，不同的Plan对应着不同的PlanNode类，构造Executor时传入PlanNode作为参数，提供必要的信息。

4. Optimizer

初步的查询计划由Optimizer修改优化。主要有两种实现方式：

  • （bustub实现）Rule-based：遍历查询计划，根据已定义好的一系列规则对PlanNode进行修改、聚合等操作。Optimizer不需要知道数据的具体内容，只根据预定义规则修改。
    • 即：每一条优化规则对应一个函数，对查询计划按一定顺序依次应用所有优化规则函数。
  • Cost-based：先读取树，利用统计学模型预测不同形式但结果等价的查询计划的cost，最后选取cost最小的查询计划。

一般来说，Planner生成Logical Plan Node，代表抽象的计划；而Optimizer生成Physical Plan Node，代表具体的计划。

抽象计划中，只有Join；具体计划中，会被优化成具体的HashJoin或NestedIndexJoin。

而在bustub中，不区分抽象与具体的Node，Planner直接生成具体计划。

5. Executor

本lab主要实现的内容，即为一系列算子。

遍历查询计划树，将树上的PlanNode替换成Executor。

算子的执行模型大致分为三种：

  • Iterator Model/火山模型：每个算子都有 Init()与 Next()两个方法。Init()初始化算子，Next()向下层算子请求下一条数据。当 Next()返回false时，表示下层算子没有剩余数据，迭代结束。
    • 一次调用请求一条数据，占用内存少，但函数调用开销大，特别是虚函数调用，容易造成cache miss问题。
  • Materialization Model：所有算子立即计算出所有结果并返回。
    • 与Iterator相反，数据量大时内存占用很高，但减少了函数调用开销。
  • Vectorization Model：上面两种模型的中和，一次调用返回一批数据。

bustub采用火山模型。

火山模型诞生初期，瓶颈是IO，且数据量不大，所以CPU占用影响不大；现在IO速度提升巨大，并且大数据时代，CPU占用问题逐渐暴露。

  • 虚函数开销：executor的next一般通过虚函数时先
    • 调用多一次虚表跳转，额外开销
    • 调用会导致简介分支预测，预测失败导致流水线被冲刷，重新获取指令、译码
  • CPU Cache利用率：每次只取一条数据，导致取的其他数据可能马上被其他算子调用的数据冲刷，不能很好利用缓存
    • 通过设置batch批量处理

算子执行方向也有两种：

  • Top-to-Bottom：从根节点算子开始，不断pull下层算子数据。
  • Bottom-to-Top：从叶子节点算子开始，向上层算子push自己的数据。

bustub采用Top-to-Bottom。

6. 获得结果

最终，在根节点获得想要查询的数据，一次sql查询结束。

7. 与前两个lab的关联

某些算子执行时，需要遍历table或索引（根据Lab 2的B+树实现），一个table由许多page组成，访问page时用到了Lab 1实现的Buffer Pool。

8. 一些不需要实现的辅助数据结构

1. ExecutorContext

初始化算子时传入，提供算子执行时的上下文。

作用为提供一系列数据结构，如Transaction、Catalog、BufferPoolManager等。

2. Catalog

通过ExecutorContext的GetCatalog()获得。

作用为存储一个数据库的元数据，包括表名、索引、视图、用户权限等信息。

3. TableInfo

通过Catalog的GetTable()获得。

作用为存储一张表的元数据，包括schema、name、table_heap与OID。

4. Schema

规定一张表存储数据的格式，即表头的元数据，包括每一列的名字、存储的数据类型等。

4. TableHeap

记录在TableInfo结构中。

作用为表的增删改查以及回滚删除。

可能由多个table page组成，但只保存第一个table page的id，访问某个page时通过page id经由buffer pool访问。

5. TablePage

继承自父类Page，实际存储table的数据。

在data开头存放next page id与prev page id，将多个table page连成双向链表，便于整表遍历。

6. TableIterator

TableHeap的迭代器。对table按行遍历。

值为Tuple。

7. Tuple

table中的一行数据，由RID唯一标识。

由value构成，value个数和类型由table info中的schema指定。

8. IndexInfo

维护一个索引表的所有元数据，如该索引所在表的名称、索引名称等。

    • 如果计算结果为true，表示符合筛选条件，返回true。
    • 否则，继续循环。

3. 实现

bustub使用的是火山模型：

  • 每个算子都有 Init()与 Next()两个方法。
    • Init()初始化算子。
    • Next(Tuple *tuple, RID *rid)向下层算子请求下一条数据。当 Next()返回false时，表示下层算子没有剩余数据，迭代结束。
  • 一次调用请求一条数据，占用内存少，但函数调用开销大，特别是虚函数调用，容易造成cache miss问题。

所以，以下每个算子的实现都分为 Init()与 Next()两个方法。

1. SeqScan

功能：顺序扫描数据表。

添加的数据结构：

  • TableInfo *table_info_：通过exec_ctx_获得catalog，然后通过plannode获得要查找的表id，接着根据表id在catalog中查找到表的元数据。
  • TableIterator table_iter：要查找的表的迭代器。

Init()：

  • 空。因为数据结构都在构造函数中初始化了。

Next(Tuple *tuple, RID *rid)：SeqScan是最底层的算子，所以只需要向指针参数提供数据。

  • 判断迭代器是否已经到达终点，如果是则返回false。
  • 取出迭代器当前指向的元组tuple与rid，传给参数。
  • 迭代器自增。
  • 返回true。

2. Insert

功能：向数据表中插入一条元组。

添加的数据结构：

  • TableInfo *table_info_
  • std::unique_ptr<AbstractExecutor> &&child_executor_：子节点的指针。因为需要插入的是子节点传过来的数据。
  • std::vector<IndexInfo *> table_indexes_：保存当前表的所有索引。因为插入一条元组需要同时向该表的所有索引表都插入。通过catalog获得。
  • bool is_end_：标识插入是否结束。

Init()：

  • 调用子节点的 Init()函数。

Next(Tuple *tuple, RID *rid)：需要接收子节点传来的数据，然后插入。tuple中装的是此次成功插入的tuple数目。

  • 如果结束标识为true，返回false。
  • 调用子节点的Next()获取下层传入的tuple、rid，如果返回true表示有数据，进入循环：
    • 在table_info_里找到table_heap结构，通过InsertTuple()插入获取的tuple。
    • 如果插入成功，根据table_indexes_对所有索引表插入tuple。
      • tuple需要根据KeyFromTuple()方法调整结构。
    • 增加count。
  • 用count构造一个tuple，传入指针参数，is_end_置为true，返回true。

3. delete

功能：在数据表中删除一条元组。删除并不是真的删除，而只是标记该数据已删除。

添加的数据结构：

  • TableInfo *table_info_
  • std::unique_ptr<AbstractExecutor> &&child_executor_：子节点的指针。因为需要删除的是子节点传过来的数据。
  • std::vector<IndexInfo *> table_indexes_：保存当前表的所有索引。因为删除一条元组需要同时在该表的所有索引表都删除。通过catalog获得。
  • bool is_end_：标识删除是否结束。

Init()：

  • 调用子节点的 Init()函数。

Next(Tuple *tuple, RID *rid)：需要接收子节点传来的数据，然后删除。tuple中装的是此次成功删除的tuple数目。

  • 如果结束标识为true，返回false。
  • 调用子节点的Next()获取下层传入的tuple、rid，如果返回true表示有数据，进入循环：
    • 在table_info_里找到table_heap结构，通过MarkDelete()删除获取的tuple。
    • 如果删除成功，根据table_indexes_对所有索引表删除tuple。
      • tuple需要根据KeyFromTuple()方法调整结构。
    • 增加count。
  • 用count构造一个tuple，传入指针参数，is_end_置为true，返回true。

4. IndexScan

功能：根据索引扫描。底层需要用到Lab2实现的B+树。

添加的数据结构：

  • IndexInfo *index_info_：索引信息
  • TableInfo *table_info_：表信息
  • BPlusTreeIndexForOneIntegerColumn *tree_index_：索引类型
  • BPlusTreeIndexIteratorForOneIntegerColumn tree_index_iter_：索引的迭代器

Init()：空。因为数据结构都在构造函数中初始化了。

Next(Tuple *tuple, RID *rid)：IndexScan是最底层的算子，所以只需要向指针参数提供数据。

  • 如果tree_index_iter_迭代结束，返回false。
  • 通过tree_index_iter_取出当前迭代索引项的rid。
  • 在表中通过GetTuple()查找到tuple信息。
    • 因为Lab2实现的都是非聚簇索引，叶子节点只存储tuple的rid，所以需要“回表”，即在原表中根据rid找到tuple具体数据。
  • tree_index_iter_前进一步。
  • 返回GetTuple()的结果，代表是否查找成功。
